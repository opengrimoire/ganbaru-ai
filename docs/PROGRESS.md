- Pomodoro timer (focus/break phases, cycle tracking, segment lifecycle, timeline rail visualization, idle detection, suspend/wake handling)
- Theming: registry-based theme store with two built-in themes plus full user-theme CRUD (create, duplicate, edit, rename, delete, plus `replaceTheme` for in-place JSON edits). User themes persist as a normalized SQLite snapshot across `themes`, `theme_tokens`, `theme_event_palette`, `theme_seed_tokens`, `theme_seed_event_palette`, and `theme_upgrade_dismissals`. Each theme stores a six-color `sources` palette (canvas, ink, primary, destructive, confirm, warning), a full resolved-token snapshot for every app and calendar shell variable, a per-token `isolated` flag, a `derivation_engine_version` stamp, plus seed mirrors of every column for per-row and "Reset all" reset. Built-in light and dark stay code-pinned (the schema's `CHECK (id NOT IN ('light', 'dark'))` defends against any import shadowing them). The boot sequence in `main.ts` awaits `ensureConfigLoaded()` and then `hydrateUserThemes()`, which runs an idempotent one-time migration that walks the legacy `themes.user` blob from `vault/config.json` through `validateThemeJson`, runs the current derivation engine to produce missing tokens, writes one transaction per theme, then strips `themes.user` from the config. JSON serves only as import/export interchange: v1 exports carry `schemaVersion: 1` plus the full snapshot, while legacy imports re-derive at import time and map the old `appTokenOverrides` / `calendarTokenOverrides` into the new isolated-flag sets. The derivation engine itself is unchanged: it is calibrated end-to-end against the dark built-in so feeding its sources through `deriveAppTokens` / `deriveCalendarTokens` reproduces `BASE_APP_TOKENS.dark` and `BASE_CALENDAR_TOKENS.dark` within a 2 rgb-unit tolerance, every formula is a signed OKLab ΔL delta off canvas or a `walkFraction(anchor, paired-bg, fraction)` between a per-surface `pickReadableForeground` anchor and that surface, calendar canvas is auto-derived from `canvas` via a direction-aware ΔL offset (-0.089 on dark, +0.032 on light) and stays pinnable through the `--cal-bg` isolated flag, and foregrounds split across `pickReadableForeground` (AA 4.5:1 on body-text pairs), `pickBrightForeground` (status/destructive/sidebar signal colors), a handful of hardcoded tokens (`--status-tentative-foreground` at `#FFFFFF`, `--form-indicator` at ink), and a calibrated OKLab-L shift on `--action-confirm-foreground`. The editor follows a buffer model: every mutator (source edit, isolate flip, link back, palette slot, rename, rebake, preset apply, per-row reset, "Reset all") updates the in-memory `UserTheme` in `$state` and re-paints the DOM but never touches SQLite during the session. Save and apply flushes the buffered theme through `persistThemeToDb`, which calls `insertTheme` for a fresh theme (created or duplicated this session) or `replaceThemeContent` for an existing one (UPDATE parent, DELETE+INSERT children, dismissals preserved). Back to themes drops the buffer: a fresh theme is removed entirely from the registry, an existing theme rolls back to the JSON snapshot captured when the editor opened. Editing a source value rebuilds the snapshot through `deriveAppTokens` and `deriveCalendarTokens`, leaves isolated tokens alone, and updates `blendCanvas` whenever `--cal-bg` participates so the dimmed event-tile blend stays in sync, all in memory until commit. `Isolated edit` flips the flag without touching the value, `Link back` re-runs derivation for that key and unflips, and "Reset all" restores every source, isolated flag, snapshot, palette slot, and blend canvas from the seed mirrors in one in-memory pass. Six tokens live in the shell to make every hardcoded `text-white` contrast-pickable: `--destructive-foreground`, `--action-danger-armed-foreground`, `--status-accepted-foreground`, `--status-tentative-foreground`, `--status-declined-foreground`, and `--pomodoro-idle-timer`; hardcoded foregrounds on buttons, badges, title bar close, event panel, meeting statuses, kanban priority, and calendar month-view day numbers all consume the token-driven or contrast-picked equivalents. `--foreground` is contrast-picked against canvas (no longer identity-equal to ink) and `--card-foreground` derives the same way against the card surface so popovers stay legible even when canvas and ink diverge. Managed from a single Appearance settings page that lists every theme with per-row Apply, Duplicate and edit, View or Edit, and Delete actions above the font and zoom controls. Every open-for-edit path activates the edited theme and hands off to a floating, draggable editor panel: the Settings modal closes, the panel mounts without a backdrop, a chevron in the header collapses the body, and the footer exposes Back to themes (rollback through `themeEditor.svelte.ts`, including delete of session-minted clones), Reset all, and Save and apply. Clicking "New theme" opens a preset picker (Sunrise, Graphite, Sepia, Nordic, High-contrast Dark, Lavender Pastel; "Start blank" keeps the seed-from-active path). The editor body renders every group inline with no mode selector. The editor surfaces an inline rebake banner whenever a loaded theme's `derivationEngineVersion` trails the code constant `DERIVATION_ENGINE_VERSION` and no dismissal row exists for the (theme, current version) pair: clicking Rebake re-runs derivation for non-isolated tokens, stamps the new version, and updates `blendCanvas` if `--cal-bg` is non-isolated, all in the buffer until commit; clicking Maybe later records the dismissal in memory and writes the `theme_upgrade_dismissals` row immediately for an existing theme, or queues it in `pendingDismissals` so it lands right after the parent insert when a fresh theme is committed. The `.dark` class on the HTML root tracks the actual canvas luminance (`isThemeDark` / `isThemeCalendarDark`) and event-tile palette selection consumes the same luminance check; the sun/moon icon is now a separate, purely decorative `scheme` tag (`light` or `dark`) that user themes can flip with one click in the editor header (built-ins peg `scheme` to their `base`). The flip is persisted on the theme (DB columns `themes.scheme` / `themes.seed_scheme` added in migration v5, JSON v1 export carries a `scheme` field, legacy imports and rows with NULL scheme default from canvas luminance via `defaultSchemeFromCanvas` and are backfilled on hydrate). Every pair row shows a live contrast indicator with per-pair targets (4.5:1 body, 3:1 recessed) and an auto-fix wand, a persistent contrast-summary bar reports total failing pairs with Jump-to-next and Fix-all, the in-house HSL color picker drives every editable field and an editable JSON panel that commits through `replaceTheme` (id locked), and JSON import/export works through both clipboard and native file dialog.
- Vault foundation: `vault/config.json` on disk with atomic write (tmp + fsync + rename) via Rust commands `vault_read_config` / `vault_write_config`. Frontend bridge (`lib/vault/config.ts`) caches the config, exposes synchronous reads, and debounces writes. One-time migration moves the legacy localStorage keys (`ganbaruai-theme`, `ganbaruai-font-family`, `ganbaruai-font-scale`) into the vault. Active theme id, font family, and font scale persist through the vault; user themes have moved to SQLite (a one-time migration on first DB-aware boot copies any remaining `themes.user` blob into the new tables and removes it from config). Two extra commands `vault_read_text` / `vault_write_text` back the dialog-driven import/export of theme files.
- Settings modal (`SettingsModal.svelte`) opens from the title-bar gear icon and hosts a single Appearance section that combines the theme list, font family, and three steppers (text size, app zoom, calendar zoom). Popovers inside the modal (color picker, font family dropdown) portal under `<body>` and position themselves with `position: fixed` so they escape the modal's `overflow` ancestors. The theme editor itself lives in a separate floating panel (`FloatingThemeEditor.svelte`) mounted at the TitleBar level: a small `themeEditor.svelte.ts` store holds the editing theme id, an effect inside the Settings modal watches it and dismisses the modal whenever an editor opens so the app beneath stays clickable, and the TitleBar renders the panel whenever the id is set. While the editor is open the TitleBar disables the theme toggle, reset-database, help, and settings buttons so they can't disrupt the edit session; window controls, pomodoro, and the performance monitor stay live. Webview zoom lives in a single `zoom.svelte.ts` store; the calendar header zoom controls and the title-bar reset-zoom button moved into the modal (keyboard shortcuts preserved).
