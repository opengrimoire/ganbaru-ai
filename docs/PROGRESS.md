Layered token model: a six-color `sources` palette (canvas, ink, primary, destructive, confirm, warning) feeds a derivation engine that tints the rest of the shell automatically. Derivation is calibrated end-to-end against the dark built-in: feeding its sources through `deriveAppTokens` / `deriveCalendarTokens` reproduces `BASE_APP_TOKENS.dark` and `BASE_CALENDAR_TOKENS.dark` exactly on source-pinned and hardcoded tokens, and within a 2 rgb-unit tolerance (~0.005 OKLab-L) on every shift-derived or walk-fraction token. That identity is what makes editing `canvas` cascade correctly: every surface moves in lockstep because every formula is a delta off canvas or a fraction between ink and the paired surface, never a pinned value. Calendar canvas is auto-derived from `canvas` via a direction-aware OKLab ΔL offset (-0.089 on dark canvases, +0.032 on light, calibrated from the built-ins' actual gaps) so cloned themes inherit the subtle grid framing on any canvas brightness; it stays pinnable through `calendarTokenOverrides["--cal-bg"]`. Surface backgrounds use a single `APP_DERIVATION` table of signed OKLab ΔL offsets calibrated from the dark built-in's own per-token lightness diffs: card (+0.028), popover (+0.056), accent (+0.077) elevate above canvas while sidebar (-0.039) recedes below it so the title bar frames the app. Applying these deltas via `shiftPerceptualL(canvas, delta)` reproduces the dark built-in's pronounced surface hierarchy on any canvas regardless of brightness. Near-white or near-black canvases clamp the upper or lower tiers at the gamut boundary rather than saturating. Recessed tokens that BASE.dark places at specific OKLab-L fractions (muted-foreground, ring, event-panel divider and text variants, pomodoro-idle-text, cal-time-label, cal-timeline-break) are produced by `walkFraction(anchor, paired-bg, fraction)` with fractions measured from BASE.dark; a contrast-target walk cannot reproduce those hexes because BASE.dark's own contrast sits between 3:1 and 4.5:1 in ways that don't collapse to a single target. The walk anchor is not raw `ink`: it is `pickReadableForeground(paired-bg, { ink, canvas, target: 4.5 })` computed per-surface so the walk always starts from a foreground that is visible against that specific surface. On BASE.dark every app surface is dark and the picker returns `ink`, preserving identity; when the user flips canvas to white while keeping ink light the per-surface anchor flips to black, so muted captions, rings, event-panel text variants, and calendar time labels all produce dark values that stay visible against the now-light surfaces. `--form-indicator` uses the same picker against canvas instead of raw ink, so checkbox/radio dots flip with canvas automatically. Foregrounds split across three strategies: `pickReadableForeground` guarantees AA 4.5:1 on body text pairs (foreground, card, popover, primary, secondary, accent surfaces) and prefers the saturating `#000`/`#fff` endpoint over a chroma-preserving walk when both ink and canvas anchors fail; `pickBrightForeground` snaps to pure `#FFFFFF` (or ink, or `#000000`) on status/destructive/sidebar surfaces so signal colors read consistently; a handful of tokens are deliberately hardcoded (`--status-tentative-foreground` = `#FFFFFF` to match BASE's intentional low-contrast signal on bright warnings, `--form-indicator` = ink) and `--action-confirm-foreground` starts with a calibrated OKLab-L shift of `confirm` (reproducing BASE.dark's pale green `#D1FAE5` on `#065F46`) and falls back to `pickReadableForeground` only when the shift collapses below 3:1. `--cal-gridline` uses `pickReadableBorder` with a lowered 1.4:1 target so cloned themes inherit the built-in's understated gridline style instead of painting prominent 3:1 lines. Six tokens live in the shell to make every hardcoded `text-white` contrast-pickable: `--destructive-foreground`, `--action-danger-armed-foreground`, `--status-accepted-foreground`, `--status-tentative-foreground`, `--status-declined-foreground`, and `--pomodoro-idle-timer`. Hardcoded foregrounds on buttons, badges, title bar close, event panel, meeting statuses, kanban priority, and calendar month-view day numbers all now consume the token-driven or contrast-picked equivalents; the light-overlay pomodoro timer flips automatically when canvas/ink flip. A three-layer resolver (override, derived, base CSS) keeps Quick colors cohesive while letting users isolate individual tokens (50 app tokens, 10 calendar tokens). `--foreground` itself is contrast-picked against canvas now (no longer identity-equal to ink), so darkening the canvas automatically flips body text everywhere instead of stranding black text on a dark background. A sibling `--card-foreground` token derives the same way against the card surface so popovers and dialogs painted on cards stay legible even when canvas and ink diverge; every in-app `bg-card` wrapper aliases `--foreground` to `--card-foreground` inside its scope so inner children that still reference `text-foreground` or inline `var(--foreground)` pick up the right color without per-class rewrites. Clone always synthesizes sources from the resolved palette so every new user theme opens in Quick-colors mode, with explicit overrides from the source theme preserved verbatim. `cloneTheme` does NOT capture identity overrides for derived tokens: the dark-BASE-identity derivation handles that automatically, so a cloned theme tracks its sources cleanly and editing canvas cascades through every non-pinned surface without fighting pinned values. `cloneTheme` captures an input-seed snapshot (`seedSources`, `seedAppTokenOverrides`, `seedCalendarTokenOverrides`, `seedBlendCanvas`, `seedEventPalette`) that powers per-row reset: every source color and every driven token has a persistent reset icon that enables when the current value diverges from the seed and clicking it restores just that one control. The editor footer exposes "Reset all" which restores every source, override, palette slot, and blend canvas from seeds in one update. Legacy themes loaded from disk without seeds run through `synthesizeSeedsIfMissing` on load (idempotent, sampled from sources or resolved tokens) so reset works uniformly. Seeds persist in the vault but are stripped from exported JSON. Managed from a single Appearance settings page that lists every theme with per-row Apply, Duplicate and edit, View or Edit, and Delete actions above the font and zoom controls. Every open-for-edit path (New, Duplicate and edit, Edit) activates the edited theme on the spot and hands off to a floating, draggable editor panel; the Settings modal closes, the panel mounts without a backdrop so the app stays clickable, and a chevron in the header collapses the body so only the drag handle and footer stay visible. The footer exposes Back to themes (rollback or delete session-minted clone, plus reinstate prior active), Reset all, and Save and apply; a forced close through the title-bar close dialog triggers a best-effort cancel. Clicking "New theme" opens a preset picker before the editor: six curated themes (Sunrise, Graphite, Sepia, Nordic, High-contrast Dark, Lavender Pastel) render as miniature previews painted from their own sources, each build-time validated to meet AA contrast across every derived token; "Start blank" keeps the original seed-from-active path. The editor body renders every group inline with no mode selector: every source color and every driven token is reachable, including a sourceless "Calendar surface" group whose `--cal-bg` row can be Isolated to pin a grid color that diverges from the auto-derived canvas track. Source-driven multi-row groups start collapsed so the list opens scannable. The sun/moon icon in the header and in the theme list tracks the actual canvas luminance (via `isThemeDark` / `isThemeCalendarDark` in `stores/themes.ts`) rather than the stored `theme.base` label, so crafting a dark theme is just a matter of darkening the canvas source. The `.dark` class on the HTML root and the event-tile palette selection both consume the same luminance check, so a mid-edit canvas flip immediately inverts every luminance-driven decision without waiting on a label toggle. `--cal-today-circle` / `--cal-today-circle-text`, `--cal-current-time`, and `--cal-timeline-focus` fall through to BASE CSS because their colors carry hard-coded semantic meaning (today's blue/gray, now's red, focus's green) that does not reduce to the source palette; everything else in the calendar is in the derivation too, so nothing falls back to a cosmetic-base-indexed default when the user works on a sourced theme. Every pair row shows a live contrast indicator checked against a per-pair target: body-text pairs target AA 4.5:1, while intentionally recessed pairs (today the muted surface, which covers hint text and past-day numbers in MonthView) target AA-large 3:1 so the warning panel respects design intent instead of false-positiving on captions designed to fade. A failing pair surfaces an amber ratio chip and a wand auto-fix button that calls `pickReadableForeground` with that row's target and writes the result as an override. A persistent contrast-summary bar above the groups reports the total number of failing pairs across every collapsed group using each row's own target, with a "Jump to next" button that cycles through failing rows (expanding the enclosing group as needed) and a "Fix all" button that runs the per-row auto-fix on every failing pair at once, so warnings produced by a distant edit cannot hide behind an unopened accordion. Editor panel chrome tracks the live user theme rather than shadowing to stable chrome values, so editing canvas gives immediate visual feedback on the panel itself; the contrast-aware foreground picker (with endpoint preference) keeps text legible even when canvas sits close to ink. The only fixed base-only tokens that remain are `--editor-chrome-thumb-border` (SV/hue/alpha slider thumb outlines) and `--editor-chrome-checker-a` / `-b` (the conic-gradient alpha checker), both of which must stay visible against any gradient the picker renders. `ColorField` accepts 8-digit `#rrggbbaa` so alpha-bearing tokens (panel edge/shadow, description editor tint, drag preview border) edit cleanly. Built-ins open in the same panel in read-only mode (palette preview and copy or save JSON panel). The in-house HSL color picker drives every editable field and an editable JSON panel whose Apply validates and commits through `replaceTheme` (id locked). JSON import/export works through both clipboard and native file dialog (tauri-plugin-dialog).
- Vault foundation: `vault/config.json` on disk with atomic write (tmp + fsync + rename) via Rust commands `vault_read_config` / `vault_write_config`. Frontend bridge (`lib/vault/config.ts`) caches the config, exposes synchronous reads, and debounces writes. One-time migration moves the legacy localStorage keys (`ganbaruai-theme`, `ganbaruai-font-family`, `ganbaruai-font-scale`) into the vault. Active theme, user themes, font family, and font scale all persist through the vault now. Two extra commands `vault_read_text` / `vault_write_text` back the dialog-driven import/export of theme files.
- Settings modal (`SettingsModal.svelte`) opens from the title-bar gear icon and hosts a single Appearance section that combines the theme list, font family, and three steppers (text size, app zoom, calendar zoom). Popovers inside the modal (color picker, font family dropdown) portal under `<body>` and position themselves with `position: fixed` so they escape the modal's `overflow` ancestors. The theme editor itself lives in a separate floating panel (`FloatingThemeEditor.svelte`) mounted at the TitleBar level: a small `themeEditor.svelte.ts` store holds the editing theme id, an effect inside the Settings modal watches it and dismisses the modal whenever an editor opens so the app beneath stays clickable, and the TitleBar renders the panel whenever the id is set. While the editor is open the TitleBar disables the theme toggle, reset-database, help, and settings buttons so they can't disrupt the edit session; window controls, pomodoro, and the performance monitor stay live. Webview zoom lives in a single `zoom.svelte.ts` store; the calendar header zoom controls and the title-bar reset-zoom button moved into the modal (keyboard shortcuts preserved).
